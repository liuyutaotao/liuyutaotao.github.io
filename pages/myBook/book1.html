# 人人都是产品经理（纪念版）

作者：苏杰

出版时间：2016-01-01

# 那些产品人的话（摘录）

  1. 无论在哪个公司，不断地对用户、对需求、对功能、对流程进行思考与总结，是成为一个成功的PM必不可少的。
  1. 产品经理是一类人，而不仅仅是一个岗位。任何人，只要能够发现问题并描述清楚，转化为一个需求，进而是一个任务，争取到支持，发动起一批人，将这个任务完成，并持续不断以主人翁的心态去维护，跟踪这个产物，那么，这个人就是产品经理。
  1. 在一个以IT及软件改变生活的时代，作为一个产品经理不仅仅改变的是一个软件产品的功能及设计，更重要的是你的产品改变的是一群人的生活方式、工作方式、认知方式、思维方式，最终达到我们的最终理想。产品经理的终极目标，就是产品让生活更美好。
  1. 技术其实只是一种手段，它最终是要转化为产品为用户服务的。
  1. 产品经理在互联网公司里面的位置很特殊，他类似于足球比赛里的中场核心队员，连接着市场、运营、销售、技术、管理层。
  1. 产品经理是有热情、爱学习的一群人！
  1. 产品是互联网的灵魂，产品是网民的精神食粮，好的产品能改变人们的生活，甚至是一个时代。

# 写在前面（摘录）

当看到一个观点的时候，就有冲动去寻找与之矛盾的观点，然后通过对不同观点的分析找出背后的原因，从而更全面地理解某个事物。一个人成熟的标志之一就是心中可以容纳各种不同的思想而无碍行事。

# 第一章

# 我的摘录

  1. 好产品能改变世界！
  2. Don't Make me Think！ （无需阅读说明书就能上手，让用户更省心）
  3. 产品的定义：
  • 产品的狭义概念：被生产出的物品；
  • 产品的广义概念：可以满足人们需求的载体。
  • 我的理解则更直白一点：产品就是用来解决某个问题的东西
  1. 产品这个东西，可以是有形的实物，也可以是无形的服务，多种多样。而解决问题其实就意味着满足人们的需求，这样才能产生价值。这个价值不仅要给产品的使用者，也要给产品的创造者.
  2. 我们工作中所做的产品，绝大多数都是在人们的需求，即用户目标和公司的商业目标之间寻找平衡。只考虑用户，公司无法赢利，必然死掉；只考虑商业，光想着公司得好处，用户留不住，公司也会死掉。 所以在这本书里我们说产品是什么，产品就是要同时解决用户的问题和公司的问题，一个都不能少。
  3. 宝洁公司的麦古利就成为全世界的第一位产品经理。
  4. 产品经理”这个概念，确实已经和旧有的产品经理概念不一样了。它更多地侧重产品本身“从无到有”、“从有到优”的过程，更多地涉及了产品规划、数据分析、用户研究、需求分析、功能设计、项目管理、敏捷方法等内容，而不是如传统的产品经理那样，去做已经有了产品之后需要做的诸如管理产品、推广和营销产品的事情。
  5. 互联网、软件行业的产品经理更重视产品功能本身的规划，需要“对市场发展趋势有敏锐的洞察力和创新意识及良好的分析、研判能力”，要能不断改进产品，要“深入了解业务，挖掘用户的多种需求，不断推出有竞争力的产品”，“制定所负责产品线的发展蓝图和实施路线图”。
  6. 在互联网、软件行业中，产品经理能真正体会到“用户是上帝”的感觉，辛辛苦苦做一个产品，给用户免费用，还要尽量让用户免费用得比付费的都爽。
  7. 管理的能力，其实就是“在资源不足的情况下把事情做成”的能力，这里的资源在产品经理的工作中通常表现为以下几种形式。
  • 第一，信息不足以决策。
  • 第二，时间不足以安排周密的计划
  • 第三，人员不足以支持工作强度和难度。
  • 第四，资金不足以自由调配。
  1. 其实，你的灵魂深处也有一个“大卫”，而你就是自己的米开朗基罗。如果把每个人的一生看做一个产品，那么你已经设计了至少二三十年，一锤又一锤，在召唤着你灵魂深处的那个“大卫”——产品经理，没错，他是原先就存在的，并不是谁刻意雕琢出来的，我想和你一起唤醒他。
  2. 应聘产品经理这类职位主要看面试，要是我来做面试官，最在乎的是应聘者有没有激情，是否够机灵、好学，逻辑思维是否清晰，沟通表达是否顺畅等。
  3. 面对那么多极具诱惑的产品，果断地放弃变得很重要，而放弃的原则与依据，就是价值观、战略这些内容。
  4. 我也开始思考什么是产品经理的灵魂，到底哪些技能之外的基本修养对一个出色的产品经理最重要？现阶段我对此给出了四个关键短语：爱生活，有理想，会思考，能沟通。

# 我的想法

第一章向读者呈现了一下几个问题：

# 1. 为什么要做产品经理？

我认为这是一个很棒的问题，当我们做一件事，或做一个决定的时，必定是有原因的。当我们自己想清楚并找到根本原因之后，随之而来的可能就是更加坚定的去前进。为什么要做产品经理呢？或许是生活中存在太多不怎么合理的东西待优化，或许是生活中的许多问题待解决，又或者想要让未来的生活更美好，改变世界...这些单是听起来都觉得非常伟大，如果有人做到了这其中的哪怕一件，我都会用45度仰望他。

😀我想做到这些，这些也是我的终极目标，是我选择做产品经理的源动力。

豪言壮语

做一个改变世界的人——产品经理。🙈

# 2. 我可以做产品经理吗？

产品的概念很广泛，包含有形的实物、无形的服务等，互联网产品经理与传统行业的产品经理有很大的不同，我们的产品是虚拟物品，生命周期也只有几个月，盈利模式也从单一卖产品变为多元盈利的模式，软件、产品很多都是给用户免费使用的，利用用户的注意力赚取第三方广告费等。在互联网、软件行业中，更加重视用户体验，产品经理能真正体会到“用户是上帝”的感觉，辛辛苦苦做一个产品，给用户免费用，还要尽量让用户免费用得比付费的都爽。需要的各方面的能力，通常还会面临各种资源不足的情况，例如：信息不足以决策；时间不足以安排周密的计划；人员不足以支持工作强度和难度；资金不足以自由调配等等。凡是资源，总会不足，既然不足，就要学会分配资源、资源管理等。总而言之，产品经理会面临各种问题，但仍要保持工作激情，对生活的热情，同时还需要全方位的综合能力，如：学习能力、洞察力、逻辑思维能力、分析能力、管理能力、沟通能力、抗压能力、同理心、共情力等等。所以产品经理真的不是一个很容易就能胜任的岗位，挑战多多呦，不过我相信我可以！

我的宣言

✌️我要做自己的米开朗琪罗，一锤一锤的雕刻出自己灵魂深处的"大卫"---产品经理。😄

# 3. 我真的想做，怎么入行呢？

首先确定✅自己真的想做产品，真的喜欢做产品，而不仅仅是喜欢做用户，然后就是要明确自己目前所处的位置，充分利用已有的知识结构、资源，找到一条最近的入行之路，最后就是先从本职工作找到与产品有关的是事情做尝试。

现在的我是一名Web前端开发工程师，算是从事着与产品经理想近的职业，每天也会和产品经理打交道，在做产品开发的过程中经常会遇到做着做着发现产品原型的逻辑不清晰，有些该说明的地方没有说明，需要多次询问才能确定功能，有时还会出现需求今天做了，明天突然就说要该的情况，有时候产品经理会出现项目与产品的需求搞不清楚的状况，这真的会给开发人员造成很多麻烦。这些情况的出现有很大一部分责任在于产品经理工作的混乱和考虑不周。希望有一天自己做产品经理时能避免这些问题。

# 4. 一个产品经理的-1～3岁

我想一个产品经理在-1～3岁时，最重要的事情就是多学、多看、多思考，践踏实地的做好每一个产品，积累经验，从需求管理到➡️项目管理再到➡️战略规划，一点一点的学习，让自己慢慢从只能打杂的小白蜕变为能够独当一面的产品经理。

# 第二章

# 我的摘录

  1. “食色性也”，“食”是为了生存，保证个体延续，“色”是为了繁衍，保证种族延续，这是生物（包括人）的本性，即最基本的需求。
  2. 马斯洛的需求层次理论
  • 生理需求
  • 安全需求
  • 社交需求
  • 尊重需求
  • 自我实现需求
  1. 人类生活在地球上，为什么会有各种各样的需求？那是,因为生活中存在太多的问题，从而产生了不满意，而问题就是“理想与现实的差距”，那么人类会很自然地产生“减少甚至消除这个差距”的愿望，这就产生了需求。需求的本质就是“问题”，问题的本质就是“理想与现实的差距”。
  2. 理解用户，是产品经理最重要的素质之一 .
  3. 用户是User，有时也叫做终端用户，End User，是使用产品的人；而客户是Customer，是购买产品的人、为产品付钱的人。
  4. "广义用户"，它指的是所有和产品有关的人，或者叫产品干系人，除了终端用户、各种客户，还包括你所在的公司里与这款产品有关的老板、销售人员、服务人员、技术人员等。“广义用户”是需求之源。不同的用户有重要程度之分，我们必须、也只能有所偏重。
    • UCD（user-centered design）,以用户为中心的设计。
    • BCD（boss-centered design）,以老板为中心的设计。
  6. 我们无法满足所有用户的需求，所以说， 优先满足哪些用户需要和产品的商业目标要结合起来考虑，我们无须满足所有用户的需求。
  7. 定性研究可以找出原因，偏向于了解；而定量研究可以发现现象，偏向于证实。两者都很重要，缺一不可，只定量会“以表代本”，看到问题但不知道原因，只定性会“以偏概全”，很可能被部分样本的特殊情况带入歧途。人们认知新事物的过程通常都是从定性到定量，再定性再定量，并且螺旋上升，而了解和证实也是不断迭代进化的。
  8. 用户研究方法：
  • 第一轮，听用户定性地说，确定产品方向，做什么？随机抽样40个用户做访谈，据此写出需求列表。
  • 第二轮，听用户定量地说，确定需求优先级，先做什么？投放了20万份调查问卷，确定了需求优先级的排序。
  • 第三轮，看用户定性地做，要先做的那几个需求，应该怎么做？一边设计，一边陆续找了10个用户来验证，做可用性测试。
  • 第四轮，看用户定量地做，根据产品的用户使用情况做数据分析，不断改进产品。
  1. 我们做用户研究的目标是：坚决杜绝“经组织研究决定，用户需要一个××功能” 这类事情发生，要实实在在地把用户当做需求之源。
  2. 需求采集过程：

明确目标➡️选择采集方法➡️制定采集计划➡️执行采集➡️资料整理➡️需求分析（下阶段）

  1. 需求采集方法：

    "需求采集人人有责"
  2. 用户访谈：
  • 用户“说”与“做”不一致。 我们要注意区分用户说的事实与观点，一般来说，诸如“我做了什么，步骤如何，碰到了什么问题”这类事实的可信度更高一些，而“我觉得、我认为”这类的观点，则需要带着大大的问号去听，并且要努力地去探究支撑观点的事实。
  • 样本少，以偏概全。 以增量的方式做访谈。举个例子，我会先访谈5个用户，得出基本结论，然后再访谈5个，观察结论是否有改变，如果有改变，就继续加大样本量，或者思考问题是否合适？样本集是否合适？如果没有改变，就停止继续访谈，节省成本。
  • 用户过于强势，把我们往沟里带。 要解决这个问题，需要时刻牢记访谈的目的。如果发现话题不对，就赶紧往正道上扳，若发现多次都扳不过来，就可以考虑尽快结束了，用户很多，不要在一个不合适的对象上花费太多时间。😝当然，有时候用户侃得十分精彩，如果你不是很忙的话，听听长长见识也可以，这个就自行把握吧。
  • 我们过于强势，把用户往沟里带。 这个问题的对策，同样是牢记访谈的目的，并且管好自己的嘴。

# 👉用户访谈的注意点：👈

首先关注目标，任务其次：比用户行为更重要的是行为背后的原因，多问问用户为什么这么做。

避免让用户成为设计师：听用户说，但不要照着做，用户的解决方案通常短浅、片面。

避免讨论技术：特别是碰到一些略懂技术的用户，不要与其纠缠产品的实现方式。

鼓励讲故事：故事是最好的帮助设计师理解用户的方法。

避免诱导性的问题：典型的诱导问题是“如果有××功能，你会使用吗？”一般来说用户会给出毫无意义的肯定答复。
